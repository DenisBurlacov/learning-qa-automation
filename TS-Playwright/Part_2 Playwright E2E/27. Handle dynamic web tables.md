## Зачем это нужно тестировщику

Представь что ты тестируешь мониторинг серверов — таблица с нагрузкой на CPU обновляется каждые 5 секунд. Строки меняются местами, колонки тоже. Ты не можешь написать `rows.nth(2).locator("td").nth(3)` потому что через секунду Chrome будет в другой строке, а CPU — в другой колонке.

Или таблица заказов с 500 записями разбита на 50 страниц. Нужно проверить что конкретный заказ есть в системе.

Это реальные задачи automation QA — и именно здесь статический подход ломается.

## Что такое динамическая таблица

**Динамическая таблица** — таблица где:

- Порядок **строк** меняется при каждом обновлении страницы
- Порядок **колонок** меняется при каждом обновлении страницы
- Значения в ячейках меняются в реальном времени

Пример: таблица процессов браузера (Chrome, Firefox, System, Internet Explorer). При каждом рефреше Chrome может быть в строке 1, 2, 3 или 4. CPU-колонка может быть первой, второй, третьей или четвёртой.

```
Было:            Name    | CPU  | Memory | Network | Disk
После рефреша:   Name    | Memory | Disk | CPU | Network
```

**Первая колонка (Name) всегда фиксирована** — это точка входа. Всё остальное плавает.

**Ключевое правило:** никогда не используй жёсткие индексы `nth(2)` или `nth(3)` для динамических данных — они будут неверными после следующего обновления.

## Стратегия работы с динамической таблицей

Задача: найти CPU-нагрузку процесса Chrome.

Проблема: Chrome может быть в любой строке, CPU — в любой колонке.

Решение — двухшаговый подход:

**Шаг 1:** Обойти все строки. В каждой строке прочитать первую ячейку (Name). Когда нашли "Chrome" — перейти к шагу 2.

**Шаг 2:** В найденной строке найти ячейку с символом `%` (это и есть CPU). Прочитать её значение.

**Шаг 3:** Сравнить захваченное значение с эталоном из жёлтой метки на странице.

Почему ищем по `%`? Потому что значения разных колонок имеют разные единицы измерения:

- CPU: `8.1%`
- Memory: `1024MB`
- Network: `3.4Mbps`
- Disk: `100MB`

Процент есть только в CPU-колонке — это уникальный признак.

## Шаг 1 — Найти нужный процесс в таблице

```typescript
import { test, expect, Locator } from '@playwright/test';

test("Verify Chrome CPU load in dynamic table", async ({ page }) => {
    await page.goto("https://practice.expandtesting.com/dynamic-table");

    // Захватываем всю таблицу
    const table: Locator = page.locator("table.table tbody");
    await expect(table).toBeVisible();

    // Конвертируем все строки в массив локаторов
    const rows: Locator[] = await table.locator("tr").all();
    console.log("Строк в таблице:", rows.length); // 4
    expect(rows).toHaveLength(4);

    // Переменная объявлена ВНЕ цикла — чтобы использовать после него
    let cpuLoad = '';

    // Обходим каждую строку
    for (const row of rows) {
        // Читаем первую ячейку (Name — всегда первая колонка)
        const processName: string = await row.locator("td").nth(0).innerText();

        if (processName === "Chrome") {
            // Chrome найден — переходим к шагу 2
            // ...
            break; // выходим из цикла — нам больше не нужно искать
        }
    }
});
```

Заметь: `for (const row of rows)` — это возможно потому что `rows` уже является массивом (`Locator[]`) после вызова `all()`.

## Шаг 2 — Захватить значение из плавающей колонки

Когда Chrome найден, нужно получить CPU-значение. Но мы не знаем в какой колонке CPU. Используем фильтрацию по содержимому ячейки.

### Два синтаксиса — одинаковый результат

**Способ 1 — CSS с has-text (чистый CSS):**

```typescript
cpuLoad = await row.locator('td:has-text("%")').innerText();
```

**Способ 2 — Playwright-синтаксис с hasText:**

```typescript
cpuLoad = await row.locator("td", { hasText: '%' }).innerText();
```

Оба делают одно и то же: среди всех `td` в этой строке найти ту, которая содержит символ `%`.

Запомни оба варианта — на интервью могут спросить про каждый.

### Как это работает

```
Строка Chrome:  | Chrome | 8.1% | 1024MB | 3.4Mbps | 100MB |
                            ^
                    td:has-text("%") — нашёл эту!
```

Локатор `td:has-text("%")` просматривает все `td` в данной строке и возвращает ту, которая содержит `%`. Независимо от того, в какой колонке она находится.

```typescript
for (const row of rows) {
    const processName: string = await row.locator("td").nth(0).innerText();

    if (processName === "Chrome") {
        // Находим ячейку с % — это CPU, в какой бы колонке она ни была
        cpuLoad = await row.locator('td:has-text("%")').innerText();
        console.log("CPU-нагрузка Chrome:", cpuLoad); // "8.1%"
        break;
    }
}
```

## CSS-метод has-text — фильтрация ячеек по тексту

Это мощный инструмент, который выходит за рамки динамических таблиц.

### Синтаксис

```typescript
// CSS-вариант
locator('tag:has-text("текст")')

// Playwright-вариант
locator("tag", { hasText: 'текст' })
```

### Примеры использования

```typescript
// Найти кнопку с текстом "Submit"
page.locator('button:has-text("Submit")')

// Найти строку таблицы, содержащую "failed"
page.locator('tr:has-text("failed")')

// Найти ячейку с процентом (любым)
row.locator('td:has-text("%")')

// Найти ячейку с MB (память)
row.locator('td:has-text("MB")')

// Playwright-вариант — то же самое
row.locator("td", { hasText: "MB" })
```

### Важное отличие от innerText

`has-text` — это **фильтр для поиска элемента**. Ты ищешь среди нескольких td ту, которая содержит текст.

`innerText()` — это **получение текста из найденного элемента**.

```typescript
// has-text НАЙДЁТ нужную ячейку
const cpuCell = row.locator('td:has-text("%")'); // ещё не читаем текст

// innerText ПРОЧИТАЕТ текст из найденной ячейки
const cpuValue = await cpuCell.innerText(); // "8.1%"

// Обычно пишем в одну строку:
const cpuValue = await row.locator('td:has-text("%")').innerText();
```

## Шаг 3 — Сравнить с эталонным значением в метке

На странице есть жёлтая метка (yellow label) которая показывает "правильное" значение CPU для Chrome. Нужно сравнить захваченное из таблицы с этим эталоном.

```typescript
// Захватываем текст жёлтой метки по ID
const yellowBoxText: string = await page.locator("#chrome-cpu").innerText();
console.log("CPU из жёлтой метки:", yellowBoxText);
// Например: "Chrome: 8.1%"

// Проверяем что захваченный cpuLoad содержится в тексте метки
// Используем toContain — потому что метка содержит "Chrome: 8.1%", а cpuLoad это "8.1%"
// toContain проверяет что строка СОДЕРЖИТ подстроку
expect(yellowBoxText).toContain(cpuLoad);
```

### toContain vs toBe

```typescript
// toBe — полное совпадение строк
expect("Chrome CPU: 8.1%").toBe("8.1%"); // FAIL — строки разные

// toContain — проверяет наличие подстроки
expect("Chrome CPU: 8.1%").toContain("8.1%"); // PASS — подстрока есть
```

В нашем случае жёлтая метка содержит весь текст типа "Chrome CPU: 8.1%", а мы захватили только "8.1%". Поэтому нужен `toContain`, а не `toBe`.

### Дополнительный вывод в консоль

```typescript
if (yellowBoxText.includes(cpuLoad)) {
    console.log("CPU-нагрузка Chrome совпадает.");
} else {
    console.log("CPU-нагрузка Chrome НЕ совпадает.");
}
```

`includes` — это JavaScript-метод строки. Возвращает `true` если строка содержит подстроку. Используется для условной логики. `toContain` — это ассерция Playwright, прерывает тест при несовпадении.

## Переменная вне цикла — зачем и почему

Это критически важный паттерн. Посмотри на объявление `cpuLoad`:

```typescript
// ПРАВИЛЬНО — объявляем ВНЕ цикла
let cpuLoad = '';

for (const row of rows) {
    if (processName === "Chrome") {
        cpuLoad = await row.locator('td:has-text("%")').innerText();
        break;
    }
}

// Можем использовать cpuLoad здесь — после цикла
expect(yellowBoxText).toContain(cpuLoad); // РАБОТАЕТ
```

```typescript
// НЕПРАВИЛЬНО — объявляем ВНУТРИ цикла
for (const row of rows) {
    if (processName === "Chrome") {
        const cpuLoad = await row.locator('td:has-text("%")').innerText();
        break;
    }
}

// cpuLoad здесь не существует — scope закрылся вместе с циклом
expect(yellowBoxText).toContain(cpuLoad); // ОШИБКА: cpuLoad is not defined
```

Это называется **область видимости (scope)**. Переменная живёт только внутри фигурных скобок `{}` где она объявлена. Если хочешь использовать значение снаружи — объяви переменную снаружи.

Также заметь: внутри цикла не используем `let` повторно — только присваиваем значение:

```typescript
let cpuLoad = '';          // объявление — один раз, снаружи
cpuLoad = await row...;    // присвоение — внутри цикла
// НЕ ПИШИ: let cpuLoad = await row... внутри цикла
```

## Полный код динамической таблицы

```typescript
import { test, expect, Locator } from '@playwright/test';

test("Verify Chrome CPU load in dynamic table", async ({ page }) => {
    await page.goto("https://practice.expandtesting.com/dynamic-table");

    const table: Locator = page.locator("table.table tbody");
    await expect(table).toBeVisible();

    const rows: Locator[] = await table.locator("tr").all();
    console.log("Строк в таблице:", rows.length);
    expect(rows).toHaveLength(4);

    // Шаг 1: найти CPU Chrome из таблицы
    let cpuLoad = '';

    for (const row of rows) {
        const processName: string = await row.locator("td").nth(0).innerText();

        if (processName === "Chrome") {
            cpuLoad = await row.locator('td:has-text("%")').innerText();
            // Альтернатива: cpuLoad = await row.locator("td", { hasText: '%' }).innerText();
            console.log("CPU Chrome из таблицы:", cpuLoad);
            break;
        }
    }

    // Шаг 2: захватить эталон из жёлтой метки
    const yellowBoxText: string = await page.locator("#chrome-cpu").innerText();
    console.log("CPU Chrome из жёлтой метки:", yellowBoxText);

    // Шаг 3: сравнить
    if (yellowBoxText.includes(cpuLoad)) {
        console.log("Совпадает.");
    } else {
        console.log("НЕ совпадает.");
    }

    expect(yellowBoxText).toContain(cpuLoad);

    await page.waitForTimeout(5000);
});
```

## Что такое пагинационная таблица

**Пагинационная таблица** — данных много, они разбиты на страницы. Одновременно видно только часть (например, 10 строк). Для просмотра остального нужно переходить на следующую страницу.

```
Страница 1: строки 1-10   [< Prev] [1] [2] [3] [4] [5] [Next >]
Страница 2: строки 11-20
...
Страница 6: строки 51-57  [Next >] — заблокирована (нет следующей страницы)
```

Три типичных задачи с пагинационными таблицами:

**Задача 1:** Прочитать данные со ВСЕХ страниц (web scraping)

**Задача 2:** Выбрать фильтр (10/25/50 строк на странице) и проверить количество отображённых строк

**Задача 3:** Найти конкретную запись через поисковую строку

## Требование 1 — Читать данные со всех страниц

### Почему while, а не for

При работе с пагинацией мы **не знаем заранее** сколько страниц. Нет фиксированного числа итераций. Это сигнал для `while`.

```typescript
// Количество страниц неизвестно — while
let hasMorePages = true;

while (hasMorePages) {
    // 1. Читаем строки текущей страницы
    // 2. Проверяем кнопку Next
    // 3. Если Next enabled — кликаем и продолжаем
    // 4. Если Next disabled — выходим (hasMorePages = false)
}
```

### Полная реализация

```typescript
test("Read all table data from all pages", async ({ page }) => {
    await page.goto("https://testautomationpractice.blogspot.com/");

    let hasMorePages = true;

    while (hasMorePages) {
        // Читаем все строки текущей страницы
        const rows: Locator[] = await page.locator("#example tbody tr").all();

        for (const row of rows) {
            // innerText() для всей строки — вернёт весь текст строки
            const rowText: string = await row.innerText();
            console.log(rowText);
        }

        // Ждём — чтобы видеть в headed-режиме
        await page.waitForTimeout(2000);

        // Проверяем кнопку Next
        const nextButton: Locator = page.locator('button[aria-label="Next"]');
        const classValue: string | null = await nextButton.getAttribute("class");

        if (classValue?.includes("disabled")) {
            // Кнопка задизейблена — больше страниц нет
            hasMorePages = false;
        } else {
            // Кнопка активна — переходим на следующую страницу
            await nextButton.click();
        }
    }
});
```

## Логика while-цикла для пагинации

Разберём пошагово что происходит:

```
Итерация 1 (страница 1):
  - Читаем 10 строк
  - Проверяем Next: class = "paginate_button next" — нет disabled
  - Кликаем Next
  
Итерация 2 (страница 2):
  - Читаем 10 строк
  - Проверяем Next: нет disabled
  - Кликаем Next
  
...
  
Итерация 6 (последняя страница):
  - Читаем 7 строк
  - Проверяем Next: class = "paginate_button next disabled" — есть disabled!
  - hasMorePages = false
  
Цикл завершается — while(false) не выполняется
```

### Почему не нужен break

В предыдущих примерах (поиск Chrome) мы использовали `break`. Здесь — нет. Почему?

При поиске Chrome: нашли — выходим из цикла немедленно через `break`.

При пагинации: когда мы делаем `hasMorePages = false`, цикл не прерывается сразу. Он завершит текущую итерацию, потом проверит условие `while(hasMorePages)` — и так как оно `false`, не войдёт в следующую итерацию. Это чище.

## Проверка кнопки Next — enabled или disabled

Это самый важный момент логики пагинации.

### Как HTML выглядит для enabled-кнопки

```html
class="paginate_button next"
```

### Как HTML выглядит для disabled-кнопки

```html
class="paginate_button next disabled"
```

Разработчики добавляют класс `disabled` к кнопке когда страниц больше нет. Это стандартная практика.

### Как захватить значение атрибута

```typescript
const nextButton: Locator = page.locator('button[aria-label="Next"]');

// getAttribute возвращает string | null
const classValue: string | null = await nextButton.getAttribute("class");
```

`getAttribute("class")` — метод Playwright для получения значения атрибута элемента. Возвращает строку или `null` (если атрибут отсутствует).

### Почему вопросительный знак перед includes

```typescript
// classValue — это string | null
// У null нет метода includes — будет ошибка!

if (classValue.includes("disabled")) { }     // ОШИБКА если classValue = null

if (classValue?.includes("disabled")) { }    // БЕЗОПАСНО: если null — вернёт undefined
```

Вопросительный знак — это **optional chaining**. Он говорит: "если значение слева от `?.` равно null или undefined — не падай, просто верни undefined".

```typescript
// Логика работы:
null?.includes("disabled")      // → undefined (не падает)
"next disabled"?.includes("disabled") // → true
"next"?.includes("disabled")    // → false
```

### Варианты локатора для кнопки Next

```typescript
// Вариант 1 — по aria-label (рекомендуется)
page.locator('button[aria-label="Next"]')

// Вариант 2 — CSS has-text (псевдоэлемент, не символ)
page.locator('[aria-controls="example"]:has-text("›")')

// Вариант 3 — nth-child
page.locator('[aria-controls="example"]:nth-child(9)')
```

## Требование 2 — Фильтрация строк через dropdown

Выбираем в дропдауне "25" и проверяем что в таблице ровно 25 строк.

```typescript
test("Filter rows and check count", async ({ page }) => {
    await page.goto("https://testautomationpractice.blogspot.com/");

    // Захватываем dropdown по ID
    const dropdown: Locator = page.locator("#example_length select");

    // Выбираем значение
    await dropdown.selectOption({ label: "25" });

    // Подход 1: через all() и length
    const rows: Locator[] = await page.locator("#example tbody tr").all();
    expect(rows.length).toBe(25);

    // Подход 2: через locator и toHaveCount (без all())
    const rowsLocator: Locator = page.locator("#example tbody tr");
    await expect(rowsLocator).toHaveCount(25);
});
```

### Разница между двумя подходами

```typescript
// Подход 1: конвертируем в массив, смотрим length
const rows: Locator[] = await page.locator("...").all();
expect(rows.length).toBe(25);
// rows.length — уже число, не Promise. Используем expect().toBe()

// Подход 2: работаем с Locator напрямую
const rowsLocator: Locator = page.locator("...");
await expect(rowsLocator).toHaveCount(25);
// toHaveCount работает с Locator. Умеет ждать (retry). Нужен await.
```

### selectOption — три способа

```typescript
// По видимому тексту (label)
await dropdown.selectOption({ label: "25" });

// По значению атрибута value
await dropdown.selectOption({ value: "25" });

// По индексу
await dropdown.selectOption({ index: 2 });
```

## Требование 3 — Поиск конкретной записи

Вводим текст в строку поиска и проверяем что запись найдена.

```typescript
test("Search for specific data in table", async ({ page }) => {
    await page.goto("https://testautomationpractice.blogspot.com/");

    // Находим поле поиска и вводим запрос
    const searchBox: Locator = page.locator("#example_filter input");
    await searchBox.fill("Paul Bird"); // запись из 4-й страницы

    // Захватываем строки после фильтрации
    const rows: Locator[] = await page.locator("#example tbody tr").all();

    let matchFound = false; // объявляем ВНЕ цикла

    if (rows.length >= 1) {
        // Есть хотя бы одна строка — проверяем содержимое
        for (const row of rows) {
            const rowText: string = await row.innerText();

            if (rowText.includes("Paul Bird")) {
                console.log("Запись найдена.");
                matchFound = true;
                break;
            }
        }
    } else {
        console.log("Строки не найдены с данным поисковым запросом.");
    }

    // Ассерция после цикла
    expect(matchFound).toBeTruthy();
    // Альтернатива: expect(matchFound).toBe(true);
});
```

### toBeTruthy vs toBe(true)

```typescript
expect(matchFound).toBe(true);      // строгое равенство: matchFound должен быть именно true
expect(matchFound).toBeTruthy();    // мягкая проверка: matchFound должен быть "правдивым"
                                     // truthy = true, 1, "слово", [] и т.д.
```

Для булевых переменных оба варианта равнозначны. На практике чаще используют `toBe(true)` для ясности.

### Зачем проверять rows.length на минимум

Поиск может вернуть 0 строк (если запись не существует). В этом случае `for...of` просто не выполнится, `matchFound` останется `false`, и тест упадёт с понятным сообщением. Проверка на минимальное количество строк даёт возможность напечатать информативное сообщение в консоль.

## Типичные ошибки и ловушки

### 1. Жёсткий индекс для динамической таблицы

```typescript
// НЕПРАВИЛЬНО — Chrome может быть в любой строке
const cpuLoad = await table.locator("tr").nth(0).locator("td").nth(2).innerText();

// ПРАВИЛЬНО — ищем Chrome, потом ищем %
for (const row of rows) {
    if (await row.locator("td").nth(0).innerText() === "Chrome") {
        const cpuLoad = await row.locator('td:has-text("%")').innerText();
        break;
    }
}
```

### 2. Переменная объявлена внутри цикла

```typescript
// НЕПРАВИЛЬНО
for (const row of rows) {
    let cpuLoad = '';            // новая переменная на каждой итерации!
    if (processName === "Chrome") {
        cpuLoad = "8.1%";
    }
}
expect(yellowBox).toContain(cpuLoad); // cpuLoad не существует здесь

// ПРАВИЛЬНО
let cpuLoad = '';               // объявляем ВНЕ цикла
for (const row of rows) {
    if (processName === "Chrome") {
        cpuLoad = "8.1%";
    }
}
expect(yellowBox).toContain(cpuLoad); // работает
```

### 3. Бесконечный while-цикл

```typescript
// ОПАСНО — если логика проверки сломается, цикл никогда не завершится
let hasMorePages = true;
while (hasMorePages) {
    // Если забыть поставить hasMorePages = false — бесконечный цикл
}

// Защита — добавить максимальное количество итераций
let hasMorePages = true;
let pageCount = 0;
const MAX_PAGES = 50;

while (hasMorePages && pageCount < MAX_PAGES) {
    pageCount++;
    // ...логика...
}
```

### 4. Забытый await у getAttribute

```typescript
// НЕПРАВИЛЬНО
const classValue = nextButton.getAttribute("class"); // Promise, не строка!
if (classValue?.includes("disabled")) { }             // всегда false

// ПРАВИЛЬНО
const classValue = await nextButton.getAttribute("class"); // строка или null
if (classValue?.includes("disabled")) { }
```

### 5. toContain vs includes — где какой

```typescript
// includes — JavaScript-метод строки, для условной логики
if (yellowBoxText.includes(cpuLoad)) {
    console.log("Совпадает");
}

// toContain — ассерция Playwright/Jest, для проверок
expect(yellowBoxText).toContain(cpuLoad); // прервёт тест если false
```

### Динамическая таблица — алгоритм

```typescript
// 1. Захватить все строки в массив
const rows: Locator[] = await table.locator("tr").all();

// 2. Объявить переменную результата ВНЕ цикла

for (const row of rows) {
    const name: string = await row.locator("td").nth(0).innerText();

        // 4. Найти нужную ячейку по содержимому (не по индексу!)
        targetValue = await row.locator('td:has-text("%")').innerText();
        break;
    }
}

const labelText: string = await page.locator("#chrome-cpu").innerText();
```
