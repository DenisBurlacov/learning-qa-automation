## Методы извлечения текста — innerText vs textContent

### Зачем это нужно

Когда тест работает с таблицей, нам нужно читать текст из ячеек. Playwright даёт два способа: `innerText()` и `textContent()`. На первый взгляд они похожи, но между ними есть важная разница.

### Как работает innerText()

Представь, что ты смотришь на страницу и читаешь то, что видишь. `innerText()` делает ровно то же самое — возвращает **только видимый текст**, точно так, как он выглядит на экране.

Что важно знать:

- Возвращает только строку (`string`) — никогда не вернёт `null`
- Убирает лишние пробелы и переносы строк
- Скрытые элементы (display: none) игнорирует

```typescript
// Получить текст конкретного продукта по индексу
const products: Locator = page.locator('.product-title');

// Взять первый элемент (индекс 0) и прочитать его текст
const productName: string = await products.nth(0).innerText();
console.log(productName); // "14.1-inch Laptop"  — чистый текст без пробелов
```

### Как работает textContent()

А теперь представь, что ты залезаешь внутрь HTML-кода и читаешь всё подряд. `textContent()` так и делает — возвращает **весь текст**, включая скрытые части, пробелы и переносы строк.

Что важно знать:

- Возвращает `string | null` — то есть либо строку, либо `null`
- Содержит лишние пробелы и переносы строк
- Читает даже скрытые элементы

```typescript
const productName: string | null = await products.nth(0).textContent();
console.log(productName);
// "\n  14.1-inch Laptop\n  "  — текст с пробелами и переносами
```

### Почему textContent требует trim()

Так как `textContent()` добавляет мусор (пробелы, переносы), нам нужно его убрать методом `trim()`. Но здесь ловушка — тип возвращаемого значения `string | null`, поэтому нужен знак вопроса:

```typescript
const productName: string | null = await products.nth(0).textContent();
console.log(productName?.trim()); // "14.1-inch Laptop"  — теперь чисто

// Знак ? перед .trim() означает: "если значение не null — вызови trim()"
// Если бы написали просто productName.trim() — TypeScript выдал бы ошибку,
// потому что null не имеет метода trim()
```

### Сравнительная таблица

||innerText()|textContent()|
|---|---|---|
|Тип возврата|`string`|`string \| null`|
|Пробелы и переносы|убирает|оставляет|
|Скрытые элементы|игнорирует|включает|
|Нужен trim()?|нет|да|
|Когда использовать|всегда, если нужен чистый текст|если нужны скрытые элементы|

### Использование в цикле

Пока `innerText()` и `textContent()` работают только с одним элементом. Чтобы пройтись по нескольким — нужен цикл. Но обычный `for...of` здесь не подойдёт, потому что `Locator` — это не массив. Только традиционный `for`:

```typescript
const count = await products.count(); // узнаём сколько элементов всего

for (let i = 0; i < count; i++) {
    const productName: string = await products.nth(i).innerText();
    console.log(productName);
}
```

Почему только традиционный `for`? Потому что нам нужен **индекс** (`i`), а `for...of` индекс не даёт. А доступ к конкретному элементу в `Locator` возможен только через `nth(i)`.

## allInnerTexts и allTextContents — для группы элементов

### Зачем нужны эти методы

Представь: у тебя 6 продуктов на странице. Тебе нужен текст всех шести. Можно написать цикл с `innerText()`. Но есть способ проще — методы `allInnerTexts()` и `allTextContents()`. Они сразу возвращают массив строк для всех элементов.

### allInnerTexts()

```typescript
const products: Locator = page.locator('.product-title');

// Получить все тексты сразу — возвращает string[]
const productNames: string[] = await products.allInnerTexts();
console.log(productNames);
// [ '14.1-inch Laptop', 'Build Computer', 'Simple Computer', ... ]
```

Возвращает `string[]` — массив чистых строк без пробелов и переносов. Можно сразу работать.

### allTextContents()

```typescript
const productNames: string[] = await products.allTextContents();
console.log(productNames);
// [ '\n  14.1-inch Laptop\n  ', '\n  Build Computer\n  ', ... ]
// — с пробелами и переносами
```

Возвращает тоже `string[]`, но с мусором. Нужно чистить через `map()` и `trim()`:

```typescript
const productNames: string[] = await products.allTextContents();

// map проходит по каждому элементу массива и применяет trim()
const trimmedNames: string[] = productNames.map(text => text.trim());
console.log(trimmedNames);
// [ '14.1-inch Laptop', 'Build Computer', ... ]  — чисто
```

### Сравнение всех четырёх методов

|Метод|Для скольких элементов|Тип возврата|Нужен trim()?|
|---|---|---|---|
|`innerText()`|одного|`string`|нет|
|`textContent()`|одного|`string \| null`|да|
|`allInnerTexts()`|группы|`string[]`|нет|
|`allTextContents()`|группы|`string[]`|да|

### Какие циклы можно использовать с массивом

Когда результат — массив (`string[]`), можно использовать **любой** вид цикла:

```typescript
const names: string[] = await products.allInnerTexts();

// for...of — самый удобный
for (let name of names) {
    console.log(name);
}

// for...in — даёт индекс
for (let i in names) {
    console.log(names[i]);
}

// традиционный for
for (let i = 0; i < names.length; i++) {
    console.log(names[i]);
}
```

## Метод all — конвертация Locator в массив

### Зачем нужен all()

Помни: у нас есть `products` типа `Locator`. С ним можно работать только через `nth(i)` и только через традиционный `for`. Но что если мы хотим удобный `for...of`?

Метод `all()` решает эту проблему. Он конвертирует `Locator` в массив локаторов (`Locator[]`). Каждый элемент массива — это локатор, указывающий на конкретный элемент страницы.

### Как работает all()

```typescript
const products: Locator = page.locator('.product-title'); // Locator — не массив

// Конвертируем в массив локаторов
const productsLocators: Locator[] = await products.all();
// Теперь это массив! Каждый элемент — локатор на конкретный продукт

console.log(productsLocators);
// [ Locator{...}, Locator{...}, Locator{...}, ... ]  — 6 локаторов
```

### Использование с for...of

```typescript
const productsLocators: Locator[] = await products.all();

for (let productLoc of productsLocators) {
    // productLoc — это локатор на конкретный элемент
    const name: string = await productLoc.innerText();
    console.log(name);
}
```

### Использование с for...in

```typescript
for (let i in productsLocators) {
    // i — это индекс (строка!), productsLocators[i] — локатор
    console.log(await productsLocators[i].innerText());
}
```

### Сравнение: Locator vs Locator[]

||`Locator`|`Locator[]` (после all())|
|---|---|---|
|Что хранит|указатель на группу|массив указателей на каждый|
|Доступ к элементу|`products.nth(i)`|`productsLocators[i]`|
|Циклы|только традиционный `for`|любой: `for`, `for...of`, `for...in`|
|Нужен await для all()?|—|да|

### Правило: когда использовать all()

Используй `all()`, когда:

- Хочешь использовать `for...of` или `for...in` вместо традиционного `for`
- Работаешь со строками таблицы и хочешь удобный перебор

```typescript
// Без all() — только традиционный for
const count = await rows.count();
for (let i = 0; i < count; i++) {
    const text = await rows.nth(i).innerText();
}

// С all() — удобный for...of
const allRows: Locator[] = await rows.all();
for (let row of allRows) {
    const text = await row.innerText();
}
```

## Виды веб-таблиц

Перед работой с таблицами важно понять: таблицы бывают разные. Playwright требует разного подхода для каждого вида.

**Статическая таблица** — содержимое не меняется. Все строки, столбцы и значения зафиксированы. Самый простой случай.

**Динамическая таблица** — данные меняются при каждом обновлении страницы. Например, мониторинг загрузки CPU браузеров: Chrome может оказаться то в первой строке, то в третьей, а процент всегда разный.

**Таблица с пагинацией** — данные разбиты на страницы. Видишь 10 строк, нажимаешь "Далее" — появляются следующие 10.

В этом конспекте разбираем **статическую таблицу**.

## Структура HTML таблицы

Прежде чем писать локаторы — нужно понять из чего состоит таблица в HTML.

```
table                   — вся таблица целиком
  tbody                 — тело таблицы (контейнер для строк)
    tr                  — одна строка (table row)
      th                — ячейка заголовка (table header)
      th
      ...
    tr                  — следующая строка
      td                — ячейка с данными (table data)
      td
      ...
```

Что важно запомнить:

- `tr` — это строка
- `th` — ячейка **заголовка** (обычно первая строка)
- `td` — ячейка с **данными** (все остальные строки)
- `tbody` — обёртка вокруг всех строк

Вот как выглядит таблица, с которой работаем в примерах (сайт testautomationpractice.blogspot.com):

```
| Book Name      | Author  | Subject | Price |
|----------------|---------|---------|-------|
| Learn Java     | Mukesh  | Java    | 500   |
| Learn Python   | Amitabh | Python  | 300   |
| Master Selenium| Mukesh  | Selenium| 2000  |
| ...            | ...     | ...     | ...   |
```

## Захват таблицы и цепочки локаторов

### Захват всей таблицы

Первый шаг — захватить таблицу как один элемент. Смотрим на атрибуты тега в DevTools и строим CSS-локатор:

```typescript
// table[name='BookTable'] — таблица с атрибутом name равным 'BookTable'
// tbody — внутри неё ищем тело таблицы
const table: Locator = page.locator("table[name='BookTable'] tbody");

// Проверяем что таблица вообще есть на странице
await expect(table).toBeVisible();
```

### Цепочки локаторов (Chaining)

Когда уже захвачен один локатор, не надо писать полный CSS снова. Можно добавить к нему ещё один шаг — это называется **цепочка локаторов**:

```typescript
// table уже захвачен выше
// Вместо page.locator("table[name='BookTable'] tbody tr")
// Пишем:
const rows: Locator = table.locator("tr"); // ищем tr внутри table
```

Это удобно, потому что:

1. Меньше кода — не повторяем `table[name='BookTable'] tbody` каждый раз
2. Если таблица поменяет атрибут — изменишь только в одном месте
3. Код читается как "из таблицы взять строки"

```typescript
// Пример цепочки из трёх шагов:
const table: Locator = page.locator("table[name='BookTable'] tbody");
const rows: Locator = table.locator("tr");          // все строки из таблицы
const headers: Locator = rows.locator("th");         // заголовки из строк

// Или так:
const secondRowCells: Locator = rows.nth(2).locator("td"); // ячейки 3-й строки
```

## Подсчёт строк и столбцов

### Подсчёт строк — два подхода

```typescript
const rows: Locator = table.locator("tr"); // захватываем все строки

// Подход 1: прямое утверждение на локатор
await expect(rows).toHaveCount(7); // ожидаем ровно 7 строк (включая заголовок)

// Подход 2: сначала считаем, потом проверяем
const rowCount: number = await rows.count();
console.log("Строк в таблице:", rowCount); // 7
expect(rowCount).toBe(7); // здесь НЕТ await — проверяем число, не Promise
```

Почему в подходе 2 нет `await` перед `expect`? Потому что `toHaveCount` работает с локатором и возвращает Promise. А `toBe` работает с уже готовым числом — никакого Promise нет.

### Подсчёт столбцов

Заголовки (`th`) — это и есть столбцы. Берём их из первой строки:

```typescript
// Заголовки находятся внутри rows (строк)
const columns: Locator = rows.locator("th"); // все th из всех строк

// Подход 1
await expect(columns).toHaveCount(4);

// Подход 2
const columnCount: number = await columns.count();
console.log("Столбцов в таблице:", columnCount); // 4
expect(columnCount).toBe(4);
```

### Пример из реального тест-прогона

```typescript
test("Проверка структуры таблицы результатов", async ({ page }) => {
    await page.goto("https://testautomationpractice.blogspot.com/");

    const table: Locator = page.locator("table[name='BookTable'] tbody");
    await expect(table).toBeVisible();

    const rows: Locator = table.locator("tr");
    await expect(rows).toHaveCount(7);

    const columns: Locator = rows.locator("th");
    await expect(columns).toHaveCount(4);
});
```

## Чтение данных из конкретной строки

### Задача

Прочитать все данные из второй строки таблицы (первая строка — заголовок, вторая строка — первые реальные данные).

### Как это работает

```typescript
// rows.nth(2) — третий элемент в наборе (индекс с нуля!)
// Индекс 0 — заголовок (th)
// Индекс 1 — первая строка данных
// Индекс 2 — вторая строка данных

const secondRowCells: Locator = rows.nth(2).locator("td");
// secondRowCells — все ячейки td из третьего tr

const secondRowTexts: string[] = await secondRowCells.allInnerTexts();
console.log("Данные второй строки:", secondRowTexts);
// [ 'Learn Java', 'Mukesh', 'Java', '500' ]
```

### Добавляем утверждение

```typescript
await expect(secondRowCells).toHaveText(["Learn Java", "Mukesh", "Java", "500"]);
```

### Печатаем каждое значение отдельно

```typescript
for (let text of secondRowTexts) {
    console.log(text);
}
// Learn Java
// Mukesh
// Java
// 500
```

### Важная ловушка: индексация с нуля

```
rows.nth(0)  — строка заголовков (th)
rows.nth(1)  — первая строка данных
rows.nth(2)  — вторая строка данных
rows.nth(3)  — третья строка данных
```

Часто путаются и пишут `nth(1)` думая что берут "первую строку", а получают заголовок. Запомни: в Playwright индексация всегда с нуля.

## Чтение всех данных из таблицы

### Задача

Прочитать все строки и все столбцы таблицы, кроме заголовка.

### Стратегия

1. Захватить все строки (уже есть в `rows`)
2. Конвертировать в массив через `all()` — чтобы использовать `for...of`
3. Пропустить первую строку через `slice(1)` — это заголовок
4. Для каждой строки взять все `td` и прочитать текст

```typescript
// Получаем массив локаторов всех строк
const allRowData: Locator[] = await rows.all();

// Печатаем заголовки отдельно (они в th, не в td)
console.log("BookName   Author   Subject   Price");

// slice(1) — пропускаем первый элемент (заголовок)
for (let row of allRowData.slice(1)) {
    // Из каждой строки берём все td и читаем текст
    const cols: string[] = await row.locator("td").allInnerTexts();
    console.log(cols.join("\t")); // \t — символ табуляции для форматирования
}
```

Вывод:

```
BookName   Author   Subject   Price
Learn Java    Mukesh    Java    500
Learn Python    Amitabh    Python    300
...
```

### Почему slice(1)?

`allRowData` содержит **все** строки включая заголовок. Заголовок имеет `th`, а не `td`. Если попробуем взять `td` из строки с заголовком — вернётся пустой массив. Поэтому пропускаем первую строку через `slice(1)`.

### Почему all() а не count() + традиционный for?

Можно написать и так:

```typescript
// Вариант с традиционным for (без all)
const count = await rows.count();
for (let i = 1; i < count; i++) { // начинаем с 1 чтобы пропустить заголовок
    const cols = await rows.nth(i).locator("td").allInnerTexts();
    console.log(cols.join("\t"));
}
```

Оба варианта работают. Но `all()` + `for...of` — читается чище и интуитивнее.

## Поиск данных по условию

### Задача

Найти все книги, автором которых является Mukesh. Напечатать их и проверить что таких книг ровно 2.

### Логика

1. Перебираем каждую строку
2. Читаем значение столбца "Автор" (индекс 1)
3. Если автор — Mukesh, запоминаем название книги (индекс 0)
4. После цикла проверяем длину массива найденных книг

```typescript
const mukeshBooks: string[] = []; // пустой массив для сбора результатов

for (let row of allRowData.slice(1)) {
    // Вариант 1: через allInnerTexts
    // const cells = await row.locator("td").allInnerTexts();
    // const authorName = cells[1];
    // const bookName = cells[0];

    // Вариант 2: читать конкретные ячейки напрямую
    const authorName: string = await row.locator("td").nth(1).innerText();
    const bookName: string = await row.locator("td").nth(0).innerText();

    if (authorName === "Mukesh") {
        console.log(`${authorName} \t ${bookName}`);
        mukeshBooks.push(bookName); // добавляем в массив
    }
}

// Утверждение: Mukesh написал ровно 2 книги
expect(mukeshBooks).toHaveLength(2);
```

### Зачем собирать в массив?

Если просто печатать — утверждение добавить некуда. Массив позволяет:

- Считать количество найденных элементов
- Проверять конкретные значения
- Использовать дальше в тесте

### Дополнительный пример: поиск тест-кейсов со статусом "failed"

```typescript
test("Найти все упавшие тесты", async ({ page }) => {
    await page.goto("https://example.com/test-results");

    const table: Locator = page.locator("table.results tbody");
    const allRows: Locator[] = await table.locator("tr").all();

    const failedTests: string[] = [];

    for (let row of allRows.slice(1)) {
        const testName: string = await row.locator("td").nth(0).innerText();
        const status: string = await row.locator("td").nth(1).innerText();

        if (status === "failed") {
            failedTests.push(testName);
        }
    }

    console.log("Упавшие тесты:", failedTests);
    expect(failedTests.length).toBe(0); // ни одного упавшего теста
});
```

## Подсчёт суммы по столбцу

### Задача

Посчитать общую стоимость всех книг в таблице. Цены находятся в четвёртом столбце (индекс 3).

### Главная ловушка: строка vs число

Данные из таблицы всегда приходят в виде строки (`string`). Нельзя складывать строки как числа — получишь "300500" вместо 800. Нужно конвертировать через `parseInt()`.

```typescript
let totalPrice: number = 0; // накопитель суммы

for (let row of allRowData.slice(1)) {
    // Читаем значение 4-й ячейки (индекс 3)
    const price: string = await row.locator("td").nth(3).innerText();

    // Конвертируем строку "500" в число 500
    totalPrice = totalPrice + parseInt(price);
}

console.log("Общая стоимость:", totalPrice); // 7100

// Утверждение
expect(totalPrice).toBe(7100);
```

### Почему parseInt, а не просто +price?

В JavaScript и TypeScript есть несколько способов конвертации:

```typescript
parseInt("500")    // 500 — безопасно, игнорирует лишние символы
parseFloat("500.5") // 500.5 — для дробных чисел
Number("500")      // 500 — строгое преобразование
+"500"             // 500 — краткая запись, но менее читаемо
```

Для цен без дробной части — `parseInt` наиболее понятен.

### Пример: проверка времени ответа API

```typescript
// Считаем среднее время ответа из таблицы результатов
let totalMs: number = 0;
let count: number = 0;

for (let row of allRows.slice(1)) {
    const responseTime: string = await row.locator("td").nth(2).innerText();
    totalMs += parseInt(responseTime);
    count++;
}

const avgMs: number = Math.round(totalMs / count);
console.log("Среднее время ответа:", avgMs, "мс");
expect(avgMs).toBeLessThan(500); // API должен отвечать быстрее 500мс
```

## Шпаргалка

### Методы получения текста

```typescript
// Один элемент — чистый текст
await locator.innerText()          // string — без пробелов, без скрытых

// Один элемент — текст с мусором
await locator.textContent()        // string | null — с пробелами и скрытыми
await locator.textContent()?.trim() // после trim — чистый текст

// Группа элементов — массив чистых текстов
await locator.allInnerTexts()      // string[] — массив без пробелов

// Группа элементов — массив текстов с мусором
await locator.allTextContents()           // string[] — с пробелами
await locator.allTextContents().then(arr => arr.map(t => t.trim())) // после trim
```

### Методы навигации по таблице

```typescript
locator.nth(0)      // первый элемент (индекс с нуля)
locator.first()     // первый элемент (то же самое)
locator.last()      // последний элемент
locator.count()     // количество элементов — возвращает Promise<number>
locator.all()       // конвертирует Locator в Locator[] — для удобных циклов
```

### Какой цикл когда использовать

|Тип данных|Какие циклы работают|
|---|---|
|`Locator`|только традиционный `for` с `nth(i)`|
|`Locator[]` (после `all()`)|`for`, `for...of`, `for...in`|
|`string[]` (после `allInnerTexts()`)|`for`, `for...of`, `for...in`|
