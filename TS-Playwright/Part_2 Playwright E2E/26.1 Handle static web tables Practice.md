## Ответы на контрольные вопросы

**1. innerText vs textContent** `innerText()` возвращает только видимый текст, без пробелов и скрытых элементов, всегда `string`. `textContent()` возвращает всё — пробелы, переносы строк, скрытые элементы, тип `string | null`. Используй `innerText` когда нужен чистый видимый текст. `textContent` — когда важны скрытые элементы.

**2. Почему trim после textContent** `textContent` возвращает "сырое" содержимое DOM-узла включая все пробелы, табы, переносы строк. `innerText` сам их убирает. После `textContent` нужен `.trim()` чтобы убрать лишние символы и получить чистое значение для сравнения.

**3. allInnerTexts — тип результата** `allInnerTexts()` возвращает промис массива строк. После await тип переменной: `string[]`.

**4. Locator vs Locator[]** `Locator` — это объект Playwright, который "помнит" как найти элемент(ы) на странице. Он не является массивом. `Locator[]` — обычный JavaScript-массив, где каждый элемент — это отдельный `Locator`, указывающий на конкретный элемент.

**5. Почему нельзя for...of для Locator** `for...of` работает с итерируемыми объектами (массивы, строки, Map, Set). `Locator` — не итерируемый объект. Нужно сначала вызвать `all()` чтобы получить `Locator[]` — обычный массив, который итерируется.

**6. Метод all()** `all()` принимает ничего, возвращает промис массива локаторов. Конвертирует локатор (Locator) в массив локаторов (Locator[]), по одному на каждый найденный элемент. Используется чтобы получить возможность использовать `for...of` и другие методы массива.

**7. for...of vs for...in для массива локаторов** `for...of` даёт сами элементы массива (в нашем случае — Locator). `for...in` даёт индексы в виде строк ("0", "1", "2"). При `for...in` нужно дополнительно обращаться `array[i]` чтобы получить элемент.

**8. Цепочка локаторов** Цепочка локаторов — это когда ты вызываешь `.locator()` на уже найденном локаторе, а не на `page`. Это лучше потому что: переиспользуешь код, поиск ограничен дочерними элементами (быстрее и точнее), не нужно каждый раз писать полный CSS-путь.

**9. await у toHaveCount но не у toBe** `toHaveCount(7)` — это ассерция Playwright на элементе. Playwright обращается к браузеру, это асинхронная операция, возвращает Promise. Нужен `await`. `toBe(7)` — это ассерция Jest/Expect на обычном числе. Число уже в памяти, никакого обращения к браузеру, не возвращает Promise. `await` не нужен.

**10. slice(1)** `slice(1)` — метод массива, возвращает новый массив начиная с индекса 1 (то есть без первого элемента). В контексте таблицы: первый элемент массива строк (индекс 0) — это заголовок с `th` тегами. Все остальные строки содержат данные (`td`). `slice(1)` пропускает заголовок.

**11. Сложение строк** В JavaScript оператор `+` для строк работает как конкатенация: `"300" + "500" = "300500"`. Чтобы сложить числа, нужно сначала конвертировать строки: `parseInt("300") + parseInt("500") = 800`.

**12. parseInt vs parseFloat** `parseInt` обрезает дробную часть и возвращает целое число. `parseInt("12.99")` вернёт `12`. `parseFloat` сохраняет дробную часть. Используй `parseInt` для цен без копеек, `parseFloat` для дробных чисел.

**13. Забытый await перед innerText** Без `await` переменная получит не строку, а объект Promise. В консоли напечатается что-то вроде `[object Promise]` или `Promise pending`. Код не упадёт с ошибкой — просто будет работать неправильно, что сложнее отлаживать.

**14. toHaveCount vs count() + toBe** `toHaveCount` — более "умная" ассерция Playwright, умеет ждать пока количество не станет нужным (retry). Полезна для динамических страниц. `count() + toBe` — более явный подход, позволяет сначала напечатать значение, потом проверить. Оба правильны, выбор зависит от задачи.

**15. HTML-теги таблицы** `table` — вся таблица. `tbody` — тело таблицы. `tr` — строка (table row). `th` — ячейка заголовка (table header), обычно в первой строке. `td` — ячейка данных (table data), в остальных строках.
